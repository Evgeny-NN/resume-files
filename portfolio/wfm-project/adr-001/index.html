ADR-001: Внедрение системы автоматизированного планирования ресурсов (WFM)

Статус: ACCEPTED
Дата: 2025
Автор: Системный аналитик
Версия: 1.0 (портфолио-версия)

1. Контекст

1.1. Проблема

В компании существует несколько контакт-центров (КЦ), обслуживающих клиентов разных бизнес-юнитов. В настоящее время:

  • Расчёт нагрузки операторов, планирование смен и составление расписаний выполняются вручную выделенными сотрудниками.
  • Мониторинг соблюдения расписания и отклонений также осуществляется вручную.
  • Штат сотрудников КЦ постоянно растёт, что делает ручное планирование неэффективным и трудозатратным.

Цель внедрения:
Автоматизировать процессы планирования и мониторинга для повышения эффективности использования ресурсов КЦ, сокращения операционных затрат и улучшения качества обслуживания клиентов.

1.2. Бизнес-требования

  • Обеспечить автоматическое прогнозирование нагрузки и построение оптимальных графиков смен.
  • Учитывать сезонные колебания, отпуска, больничные и другие факторы.
  • Предоставить гибкие инструменты для управления сменами (обмен, заявки, доработки).
  • Обеспечить мониторинг фактической занятости операторов в режиме, близком к реальному времени.
  • Интегрироваться с существующими системами: кадровым справочником (Active Directory), чат-платформой, голосовой платформой, системой управления персоналом.

1.3. Технические ограничения

  • Развёртывание в корпоративном Kubernetes-кластере (Shared Application Space).
  • Использование стандартных корпоративных сервисов: авторизация через Keycloak, потоковая передача данных через Kafka, хранение в PostgreSQL.
  • Высокая доступность (BUSINESS CRITICAL) и производительность (поддержка до 800 операторов, ~24 000 событий в сутки).

2. Архитектурное решение

2.1. Выбранное решение

Внедряется новая информационная система cs-wfm на базе коробочного решения вендора (Naumen WFM).
Решение обеспечивает полный цикл WFM: прогнозирование, планирование, мониторинг и отчётность.

2.2. Обоснование выбора

  • Вендор уже внедрял решения в компании (опыт работы с голосовой платформой).
  • Коробочное решение покрывает все функциональные требования и имеет готовые интеграционные адаптеры.
  • Возможность гибкой настройки под бизнес-процессы компании.

2.3. Архитектурные компоненты

  • Адаптер (wfm-adapter) – сервис-посредник для асинхронных интеграций:
    • Подписывается на топики Kafka для получения онлайн-статусов операторов из чат-платформы.
    • Агрегирует и преобразует данные, передаёт их в WFM через REST API.
  • Коннектор (wfm-connector) – встроенный компонент вендора для синхронных взаимодействий:
    • Запрашивает исторические данные по голосовым звонкам из БД голосовой платформы через JDBC.
    • Получает онлайн-статусы операторов голосовой платформы по WebSocket.
  • Интеграция с Active Directory (LDAPS) – регулярный импорт данных о сотрудниках (логин, ФИО, подразделение, должность) для автоматического создания/обновления карточек операторов.
  • Интеграция с Keycloak – аутентификация и авторизация пользователей WFM, а также выдача токенов для межсервисных взаимодействий.
  • Интеграция с внешним провайдером пуш-уведомлений (Google Firebase) для оповещения операторов о сменах.

4. Последствия

4.1. Позитивные

  • Автоматизация прогнозирования и планирования, снижение ручного труда.
  • Повышение уровня обслуживания клиентов за счёт оптимальной занятости операторов.
  • Оптимизация ФОТ (фонда оплаты труда) за счёт точного планирования.
  • Гибкость настроек системы под изменяющиеся бизнес-процессы.
  • Унификация с уже используемым вендорским решением для голосовой платформы.

4.2. Негативные

  • Необходимость поддержки коробочного решения, которое не полностью соответствует принятым в компании технологическим стандартам (например, настройка JDBC-подключений, отсутствие некоторых паттернов).
  • Потребность в дополнительном обучении персонала.
  • Сложность интеграции с нестандартными источниками данных (например, чат-платформа передаёт статусы через Kafka, требуется адаптер).

5. Ключевые чекпоинты

  • Создание целевого ADR и согласование.
  • Запрос инфраструктуры для тестового и продуктивного контуров.
  • Настройка окружения и установка платформы WFM.
  • Интеграция с голосовой платформой (online-статусы, история).
  • Интеграция с чат-платформой (online-статусы через Kafka).
  • Интеграция с Active Directory (синхронизация сотрудников).
  • Интеграция с Keycloak (аутентификация/авторизация).
  • Интеграция с системой управления персоналом (StaffDesk) для автоматического заведения сотрудников.
  • Проведение опытно-промышленных испытаний (ОПЭ) и передача в эксплуатацию.
